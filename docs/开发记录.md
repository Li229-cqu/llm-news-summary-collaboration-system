### 2.2 关键设计决策

| 决策 | 说明 |
|------|------|
| **Teleport to body** | 浮动按钮和弹窗用 `<Teleport to="body">` 确保 z-index 不受父元素影响 |
| **拖拽用 PointerEvent** | `pointerdown/pointermove/pointerup` + `setPointerCapture`，比 `mousedown` 更通用（支持触屏） |
| **点击 vs 拖拽区分** | 移动距离 < 5px 视为点击，≥ 5px 视为拖拽 |
| **弹窗定位** | 始终在按钮上方打开，水平居中于按钮，超出屏幕时自动左/右偏移 |
| **useRoute() 位置** | 必须放在 `<script setup>` 顶层调用，不能放在函数内部使用，否则返回 undefined |

### 2.3 历史记录存储

## 三、RAG 增强搜索系统

### 3.1 架构

```
用户提问 → 前端采集页面上下文(route)
         → POST /api/assistant/search { question, context }
         → 后端 MultiResourceRetriever 三层检索
         → 返回 { current, articles[] }
         → 前端拼接 Prompt
         → POST LLM API (127.0.0.1:8025)
         → 展示答案 + 📎 参考来源
```

### 3.2 后端文件

```
backend/app/modules/rag_assistant/
├── __init__.py      # 模块声明
├── schema.py        # SearchRequest / SearchContext / SearchResultItem
├── rag.py           # MultiResourceRetriever（原名 retriever.py，后改名）
└── router.py        # POST /api/assistant/search
```

在 `main.py` 注册：
```python
from app.modules.rag_assistant.router import router as rag_assistant_router
app.include_router(rag_assistant_router)
```

### 3.3 三层检索策略

| 层级 | 方法 | 数据源 | SQL |
|------|------|------|-----|
| **Step 0** 当前资源 | `_get_current_resource` | 根据 `context.page` 判定，查 `news` / `community_post` / `news_topic` | `WHERE id = %s` |
| **Step 1** 关联检索 | `_get_related` | 同话题新闻 + 关联帖子 + 热门评论 | `WHERE topic_id = %s` / `WHERE related_news_id = %s` |
| **Step 2** 全文搜索 | `_fulltext_multi_search` | news (FULLTEXT INDEX) + community_post (LIKE) | `MATCH...AGAINST (BOOLEAN MODE)` / `LIKE %kw%` |

### 3.4 context.page 路由判定

| 页面 | page 值 | 关键参数 | 当前资源类型 |
|------|------|------|------|
| `/news/:id` | `news-detail` | `newsId` | news |
| `/community/posts/:id` | `community` | `postId` | community_post |
| `/timeline/:topicId` | `timeline` | `topicId` | news_topic |
| `/home` | `home` | `categoryId` / `searchKeyword` | 虚拟资源 |

## 四、重要踩坑记录

### 4.1 Qwen-Lora 模型不支持 system 角色

**现象**：LLM 返回 500 `jinja2.exceptions.TemplateError: System message must be at the beginning.`

**原因**：Qwen 模型的 Jinja2 chat template 检查所有消息，发现 `role: "system"` 就抛异常。

**另外一些模型要求 system 消息必须在 messages 数组的第一位置**：如果 messages 中有 system 消息但不在 [0] 位置，也会报错。

**解决**：不使用 `role: "system"`，将指令作为纯文本拼接到 `role: "user"` 消息中。

```typescript
// ❌ 错误
[
  { role: 'system', content: '你是AI助手...' },
  { role: 'user', content: question }
]

// ✅ 正确
[
  { role: 'user', content: '【指令】你是AI助手...\n\n' + question }
]
```

### 4.2 sessionStorage 跨标签不共享

**现象**：调试 HTML 页面读不到 AI 助手写入的日志数据。

**原因**：`sessionStorage` 按浏览器标签隔离，不同标签页的 sessionStorage 互不可见。

**解决**：跨标签共享数据必须用 `localStorage`。

### 4.3 Vite proxy 解决跨域

**现象**：前端 `fetch('http://127.0.0.1:8025/...')` 被浏览器 CORS 拦截。

**解决**：在 `vite.config.ts` 配置代理：

```typescript
proxy: {
  '/llm-api': {
    target: 'http://127.0.0.1:8025',
    changeOrigin: true,
    rewrite: (path) => path.replace(/^\/llm-api/, ''),
  },
}
```

前端请求改为 `/llm-api/v1/chat/completions`，Vite 自动转发。

### 4.4 useRoute() 调用位置限制

**现象**：`Cannot read properties of undefined (reading 'path')`

**原因**：Vue 3 的 `useRoute()` 必须在 `<script setup>` 顶层同步调用，不能放在异步函数或事件处理函数内部。

```typescript
// ❌ 错误 — 在函数内调用
function buildContext() {
  const route = useRoute()  // → undefined
}

// ✅ 正确 — 在顶层调用，函数内引用
const route = useRoute()
function buildContext() {
  const p = route.path  // → 正常
}
```

### 4.5 LLM 回复带 markdown 格式

**现象**：LLM 回复开头总带 `#` 号，因为上下文中用了 `###` 标记。

**解决**：在指令中明确要求 `不要使用markdown格式（如#、**等标记），用纯文本回复。`
