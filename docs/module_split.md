# 《基于大语言模型的智能新闻摘要与协同互动系统》模块拆分说明

本文档用于明确系统后续的模块划分、职责边界与并行开发方式，供小组成员进行任务分工和协作参考。

## 一、模块独立开发原则

1. 先搭建公共框架，统一项目结构、路由、接口规范、响应格式与 Mock 数据约定。
2. 公共框架稳定后，再按业务模块并行开发。
3. 每个模块有独立的前端页面，负责自身的页面展示与交互流程。
4. 每个模块有独立的 API 文件，集中维护该模块的请求方法与数据类型。
5. 每个模块有独立的后端 `router`、`service`、`schema`，分别负责接口入口、业务逻辑与请求响应模型。
6. 当前阶段统一使用 Mock 数据，确保各模块可独立推进并降低外部依赖。
7. 后期再统一接入数据库，并逐步将 Mock 数据替换为真实数据访问实现。

## 二、前端模块划分

| 模块目录 | 模块名称 | 主要职责 |
| --- | --- | --- |
| `home` | 首页新闻模块 | 展示新闻列表、分类入口、推荐内容和热榜等首页信息。 |
| `news-detail` | 新闻详情模块 | 展示新闻正文、相关新闻及详情页互动入口。 |
| `ai-generate` | AI 标题摘要生成模块 | 提供 AI 标题与摘要生成的输入、提交和结果展示页面。 |
| `community` | 社区互动模块 | 提供帖子浏览、发布、评论、点赞等社区交互页面。 |
| `profile` | 个人中心模块 | 展示用户资料、个人记录和个人相关内容。 |
| `admin` | 管理后台模块 | 提供审核、编辑、用户及内容管理等后台页面。 |
| `login` | 登录模块 | 提供登录、注册及身份状态相关页面。 |

## 三、后端模块划分

| 模块 | 主要职责 |
| --- | --- |
| `auth` | 登录注册与身份认证。 |
| `user` | 用户信息维护与查询。 |
| `news` | 新闻列表、分类、详情、热榜。 |
| `interaction` | 点赞、收藏、评论。 |
| `ai` | AI 生成、要素抽取、质量校验、AI 问答。 |
| `community` | 发帖、帖子流、社区热搜、社区 AI 助手。 |
| `profile` | 个人中心聚合查询。 |
| `admin` | 审核/编辑与管理员后台。 |

各后端模块均应保持 `router.py`、`service.py`、`schema.py` 的清晰分层，并通过公共响应、认证与异常处理机制完成协作。

## 四、AI 服务模块划分

| 模块 | 主要职责 |
| --- | --- |
| `generate_service` | 标题摘要生成。 |
| `extract_service` | 关键词和新闻要素抽取。 |
| `check_service` | 一致性质量校验。 |
| `chat_service` | AI 新闻助手问答。 |

AI 服务在当前阶段可为上述能力提供对应的 Mock 实现，后续再按统一接口替换为真实模型调用。

## 五、模块之间的边界

1. 新闻模块不直接处理 AI 生成；需要 AI 能力时，通过后端 AI 模块调用 AI 服务。
2. AI 生成模块不直接处理社区帖子；社区内容由社区模块负责。
3. 社区模块不直接处理个人中心聚合；个人相关数据由个人中心模块统一查询与展示。
4. 个人中心只负责聚合展示，不承担其他业务模块的核心业务逻辑。
5. 后台模块只负责审核和管理，不替代前台业务模块的具体功能实现。
6. AI 服务只负责生成和分析，不直接管理用户权限；认证和权限控制由后端相关模块负责。

## 六、后续并行开发方式

在公共框架、接口规范和 Mock 数据约定完成后，新闻、AI、社区、个人中心、后台模块可以并行开发。

各模块负责人应优先确定本模块页面、接口和数据结构，并及时同步接口变更。跨模块需求通过已定义的 API 或公共能力协作，避免直接依赖其他模块的内部实现；待各模块完成阶段性功能后，再统一进行集成联调。
