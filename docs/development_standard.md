# 《基于大语言模型的智能新闻摘要与协同互动系统》开发规范

本文档用于统一小组后续开发过程中的目录组织、模块边界、命名方式、接口格式与 Git 提交习惯。

## 一、通用开发规范

1. 项目采用前后端分离架构。
2. 每个业务模块应独立开发，明确自身职责与边界。
3. 不同模块之间通过 API 通信，不直接耦合。
4. 可复用的组件、工具和通用能力应放在公共目录中，避免重复实现。
5. 页面、接口、数据结构的命名应保持统一；新增或变更约定时，应及时同步给小组成员。

## 二、前端开发规范

后续 `frontend` 目录计划使用以下技术栈：

- Vue 3
- Vite
- TypeScript
- Vue Router
- Pinia
- Element Plus
- Axios

### 前端目录规划

| 目录 | 用途 |
| --- | --- |
| `src/api` | 接口封装。 |
| `src/router` | 路由配置。 |
| `src/stores` | 状态管理。 |
| `src/layouts` | 整体布局。 |
| `src/views` | 首页、新闻详情、AI 生成、社区、个人中心、后台等页面。 |
| `src/components` | 公共组件。 |
| `src/utils` | 工具函数。 |
| `src/assets` | 静态资源。 |

### 前端命名要求

1. 页面组件使用 PascalCase 命名，例如 `NewsDetail.vue`。
2. API 文件按模块命名，例如 `news.ts`、`ai.ts`、`community.ts`。
3. 路由路径使用小写字母和短横线，例如 `/news-detail`。
4. 不同模块的页面不要互相直接 `import`；跨模块协作应通过路由、状态管理、公共组件或 API 完成。

## 三、后端开发规范

后续 `backend` 目录计划使用 FastAPI。

### 后端目录规划

| 路径 | 用途 |
| --- | --- |
| `app/main.py` | 项目入口。 |
| `app/core` | 配置。 |
| `app/common` | 统一响应、异常处理等公共能力。 |
| `app/modules` | 业务模块。 |
| `app/mock` | Mock 数据。 |

### 模块内部结构

每个业务模块原则上包含以下文件：

| 文件 | 职责 |
| --- | --- |
| `router.py` | 接口入口与路由定义。 |
| `service.py` | 业务逻辑。 |
| `schema.py` | 请求和响应模型。 |

路由层应专注于接收请求和返回响应；业务处理应放在服务层；请求与响应的数据结构应通过模型统一定义。

## 四、AI 服务开发规范

`ai-service` 独立于 `backend`，用于承载 AI 模型调用相关能力。

1. 当前阶段先使用 Mock 返回数据。
2. 后续可将 Mock 实现替换为真实大语言模型 API 或本地模型服务。
3. 前端不得直接调用大语言模型 API。
4. 前端只能调用 `backend`。
5. `backend` 再调用 `ai-service`，由后端统一管理 AI 服务调用、参数校验与结果转换。

调用链如下：

```text
frontend → backend → ai-service → 大语言模型 API / 本地模型服务
```

## 五、API 响应规范

所有接口统一使用以下成功响应结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

错误响应使用以下结构：

```json
{
  "code": 400,
  "message": "错误信息",
  "data": null
}
```

其中，`code` 表示业务状态码，`message` 表示结果说明，`data` 表示实际响应数据；无数据时使用 `null`。

## 六、Git 提交规范

建议使用以下提交前缀：

| 格式 | 说明 |
| --- | --- |
| `feat: 新增功能` | 新增功能。 |
| `fix: 修复问题` | 修复问题。 |
| `docs: 修改文档` | 修改文档。 |
| `style: 样式调整` | 样式调整。 |
| `refactor: 重构代码` | 重构代码。 |
| `test: 测试相关` | 测试相关变更。 |
| `chore: 工程配置` | 工程配置、依赖或构建相关变更。 |

提交信息应简洁说明本次变更内容；涉及接口、数据结构或公共配置变更时，应在提交说明或协作沟通中明确影响范围。
